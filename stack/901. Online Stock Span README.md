# 901. Online Stock Span

## Problem Summary

Design a class that keeps track of stock prices and returns the span of the current price.

The span of a price is the number of consecutive previous days where the price was less than or equal to the current price.

Example:
- If the recent prices are [7, 2, 1, 2] and today’s price is 2, the span is 4.

---

## 1. How to Choose the Right Approach from Constraints

Before writing code, always read the constraints first. They tell you how large the input can become, and that helps you decide whether a simple idea is good enough or whether you need a more efficient solution.

### What the constraints are telling us here

- At most 10^4 calls to next()
- Each call may need to look at many previous days

This means the input size is not tiny. Even though 10^4 looks smaller than 10^9, it is still a large number of operations for a slow algorithm.

### Why this matters

A constraint like 10^9 is only a rough upper bound for the size of input in some problems, but the important idea is not the exact number itself. The important idea is how the runtime grows as n grows.

For example, compare these growth rates:

- O(1): constant work
- O(log n): very small growth
- O(n): grows linearly
- O(n log n): grows a bit faster
- O(n^2): grows much faster
- O(2^n): becomes impossible very quickly

### Important concept: 10^4 is still large

Even though 10^4 is smaller than 10^9, it is still a big number for some algorithms.

If n = 10^4:

- O(n) means about 10^4 steps
- O(n^2) means about 10^8 steps
- O(n log n) means about 10^4 × 13 = around 1.3 × 10^5 steps

So a quadratic solution may still be too slow, even for 10^4 input, because 10^8 operations is a lot for most languages.

### Why we should not use brute force here

A brute force approach may scan backward again and again for every new price. That can lead to repeated work and a much larger total cost.

That is why we prefer a stack-based approach:

- Brute force: O(n) per call in the worst case
- Stack approach: O(1) amortized time per call
- Space: O(n) for storing previous prices and spans

### Rule of thumb for choosing an approach

- If the constraints are small, a simple solution is often fine.
- If the constraints are medium or large, avoid nested loops or repeated scanning when possible.
- If the problem asks for repeated queries, look for a data structure that can reuse previous results.
- If the problem has many operations, aim for O(log n) or O(n) rather than O(n^2).

### For this problem

Because we may call next() many times, and each call could otherwise do a lot of repeated work, the stack solution is the better choice. It is faster, cleaner, and scales much better.

## 2. Brute Force Approach

### Idea
For each new price, look backward day by day and count how many consecutive days have price less than or equal to the current price.

### Why it is slow
This approach checks many previous days repeatedly, which makes it inefficient for many calls to next().

### Brute Force Code (Java)

```java
class StockSpanner {
    private List<Integer> prices;

    public StockSpanner() {
        prices = new ArrayList<>();
    }

    public int next(int price) {
        prices.add(price);
        int span = 0;

        for (int i = prices.size() - 1; i >= 0; i--) {
            if (prices.get(i) <= price) {
                span++;
            } else {
                break;
            }
        }

        return span;
    }
}
```

### Brute Force Complexity
- Time Complexity: O(n) per call in the worst case
- Space Complexity: O(n)

---

## 2. Best Approach: Monotonic Stack

### Why we use a stack
We need to quickly find the previous greater element and also maintain the span efficiently.

A stack helps us store pairs of:
- price
- span of that price

When a new price arrives:
- while the previous price is smaller or equal, we add its span to the current span
- then we push the current price with the computed span

### Key Idea
- Keep a stack of pairs: [price, span]
- For each new price, accumulate span while the stack top price is smaller or equal
- Push the current price and span

### Why this is better
- It avoids repeatedly scanning from the start every time
- It gives an efficient O(1) amortized time per call
- It is a classic stack-based pattern

### Optimized Java Solution

```java
import java.util.Stack;

class StockSpanner {
    private Stack<int[]> stack;

    public StockSpanner() {
        stack = new Stack<>();
    }

    public int next(int price) {
        int span = 1;

        while (!stack.isEmpty() && stack.peek()[0] <= price) {
            span += stack.pop()[1];
        }

        stack.push(new int[]{price, span});
        return span;
    }
}
```

### Optimized Complexity
- Time Complexity: O(1) amortized per call
- Space Complexity: O(n)

---

## 3. Example

### Input
[100, 80, 60, 70, 60, 75, 85]

### Output
[1, 1, 1, 2, 1, 4, 6]

### Explanation
- Each call to next() returns the span for that price.
- The stack keeps track of previous smaller or equal prices efficiently.

---

## 4. Constraints

- 1 <= price <= 10^5
- At most 10^4 calls will be made to next()

These constraints make the stack solution ideal because it handles repeated calls efficiently.

## 5. Important Notes for Revision

- This problem is a great example of using a monotonic stack.
- We store the price and its span together.
- The stack allows us to reuse previous information instead of recomputing everything.
- This is a very useful pattern for interview preparation.
